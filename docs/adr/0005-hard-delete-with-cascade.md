# ADR-0005: 削除方針 — ソフトデリート不採用、物理DELETE

## Context

ファイルシステムから消えた画像のDB上の扱いとして、`deleted_at` カラムによるソフトデリートを検討した。タグやお気に入りメタデータの保護が動機だった。

## Decision

物理DELETEを使う。`ON DELETE CASCADE` でタグ関連も自動削除する。

## Alternatives

| 選択肢                                                     | 利点                                       | 欠点                                                                             | 不採用理由                                                                                                                                                          |
| ---------------------------------------------------------- | ------------------------------------------ | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `deleted_at` カラムによるソフトデリート + 将来のパージ処理 | メタデータを保護できる、誤削除から復旧可能 | 全クエリに `WHERE deleted_at IS NULL` が必要でバグの温床、パージ処理等の運用負担 | DB には画像のメタデータしかなく、ファイルが消えたらメタデータだけ残しても意味がない。お気に入りがタグに統合されたことで保護すべき独自メタデータの価値が更に下がった |

## Consequences

**Positive:**

- クエリがシンプルになる（`WHERE deleted_at IS NULL` が不要）
- パージ処理等の運用負担がない
- `ON DELETE CASCADE` で関連データも自動的にクリーンアップされる

**Negative:**

- 誤削除時にメタデータの復旧ができない（ただし画像ファイルが唯一のソースであり、ファイルがなければメタデータの価値もない）
